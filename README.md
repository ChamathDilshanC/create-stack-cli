# create-stack

The **Ultimate Multi-Tiered Project Orchestrator** ‚Äî an interactive CLI that scaffolds Frontend, Fullstack, Backend, Desktop, and Mobile projects using each stack's own official tooling. No static templates, ever: every project comes from a live `create-vite`, `create-next-app`, the Angular CLI, the Nest CLI, or similar ‚Äî so what you get is always current, never a stale copy.

## Features

- **Official scaffolders, not static templates** ‚Äî every framework is scaffolded by running its own real initializer CLI. Express and Fastify are the one deliberate exception (neither ships an official scaffolder), so those are generated by hand instead.
- **Five project types, one decision tree** ‚Äî Frontend, Fullstack, Backend, Desktop, Mobile ‚Äî each asking only the questions that make sense for it (no "styling" question for a backend API, no "database" question for a static site).
- **Styling:** Tailwind CSS v4 (official Vite-plugin setup) or UnoCSS, wired into whichever config shape the framework uses (plain `vite.config`, Nuxt/Astro's nested `vite:` key, or the framework's own official flag).
- **Database/ORM:** Prisma (`prisma init`), Drizzle ORM, or Mongoose ‚Äî installed and wired with a working starter schema/model.
- **Code quality:** ESLint + Prettier, or Biome ‚Äî mutually exclusive by design (a single choice, not two checkboxes you could contradict).
- **Docker:** an optional Dockerfile + docker-compose.yml, tailored to the project (multi-stage Node build for servers, build-then-nginx for static frontends).
- **Enterprise folder structure:** a `.gitkeep`-tracked, feature-sliced `src/` layout ‚Äî one shape for Frontend/Mobile/Desktop, another for Backend (with `schema/` swapped in for `models/` when Drizzle is chosen).
- **Auto-installs dependencies** with your package manager of choice: npm, yarn, pnpm, or bun.
- **Non-interactive mode** via CLI flags, for scripting and CI.
- Graceful handling of existing directories, network failures during install, and Ctrl+C cancellation ‚Äî no unhandled promise rejections.

## Quick start

```bash
npx @chamathdilshanc/create-stack my-app
```

Then follow the prompts:

```
Project name: my-app
What are you building? Frontend
Select a framework: React
Language: TypeScript
Styling: Tailwind CSS (v4)
Code quality tooling: ESLint + Prettier
Add Docker support? No
Install dependencies with: pnpm
Install dependencies now? Yes
```

## The decision tree

| Question | Shown for | Options |
| --- | --- | --- |
| Project type | always | Frontend, Fullstack, Backend, Desktop, Mobile |
| Framework | always | see table below |
| Language | unless the framework forces one | TypeScript (default), JavaScript |
| Styling | Frontend, Fullstack, Desktop | Tailwind CSS (v4), UnoCSS, CSS Modules, None |
| Database / ORM | Backend, Fullstack | Prisma, Drizzle ORM, Mongoose, None |
| Code quality | always | ESLint + Prettier, Biome, None |
| Docker | always | Yes / No |

Angular and NestJS force TypeScript, so the language question is skipped for them.

### Frameworks by project type

| Type | Frameworks |
| --- | --- |
| Frontend | React, Vue, Angular, Svelte, SolidJS |
| Fullstack | Next.js, Nuxt.js, SvelteKit, Astro |
| Backend | Express.js, NestJS, Fastify, Hono |
| Desktop | Electron, Tauri |
| Mobile | Expo (React Native) |

## Non-interactive usage

Every prompt can be skipped by passing the equivalent flag. Combine `--yes` with the required flags to run fully non-interactively (useful in CI):

```bash
npx @chamathdilshanc/create-stack my-app \
  --type backend \
  --framework nestjs \
  --database prisma \
  --quality eslint-prettier \
  --docker \
  --pm pnpm \
  --yes
```

### CLI options

| Flag | Description |
| --- | --- |
| `[project-directory]` | Directory to create the project in (positional argument) |
| `--type <type>` | Project type: `frontend`, `fullstack`, `backend`, `desktop`, `mobile` |
| `-f, --framework <name>` | Framework within the chosen type (see table above) |
| `-l, --language <lang>` | `ts` or `js` |
| `-s, --styling <name>` | `tailwind`, `unocss`, `css-modules`, `none` |
| `-d, --database <name>` | `prisma`, `drizzle`, `mongoose`, `none` |
| `-q, --quality <name>` | `eslint-prettier`, `biome`, `none` |
| `--docker` | Add a Dockerfile + docker-compose.yml |
| `-p, --pm <manager>` | Package manager: `npm`, `yarn`, `pnpm`, `bun` |
| `--no-install` | Skip automatic dependency installation |
| `-y, --yes` | Skip prompts; fails if a required option isn't supplied via flags |
| `--overwrite` | Overwrite the target directory if it already exists, without prompting |
| `-V, --version` | Print the CLI version |
| `-h, --help` | Print usage |

## How scaffolding works

There are no template folders. `scaffold.js` dispatches to one handler per project type, each of which runs the official initializer for the chosen framework:

- **Frontend** ‚Äî `npm create vite@latest` for React/Vue/Svelte/Solid; the Angular CLI (`ng new`) for Angular.
- **Fullstack** ‚Äî `create-next-app`, `nuxi init`, `sv create` (SvelteKit), or `create-astro`.
- **Backend** ‚Äî the Nest CLI (`nest new`) for NestJS, `create-hono` for Hono. Express and Fastify have no official scaffolder, so a clean `package.json` + `src/server.{ts,js}` are generated by hand instead.
- **Desktop** ‚Äî `create-tauri-app` or `create-electron-app` (Electron Forge).
- **Mobile** ‚Äî `create-expo-app`.

On top of whichever project that produces, the CLI layers:

1. **Styling** ‚Äî installs the right packages and wires the plugin into the framework's actual config shape. Next.js/SvelteKit already have official `--tailwind`/`--add tailwindcss` flags, so those are used directly instead of a post-hoc rewrite; Nuxt, Astro, and everything Vite-based go through a generic config injector.
2. **Database** ‚Äî `npx prisma init` for Prisma; a hand-written `drizzle.config.ts` + schema + client for Drizzle; a connection module + example model for Mongoose. SvelteKit's own `--add drizzle` addon is used instead when both are selected together.
3. **Code quality** ‚Äî ESLint+Prettier or Biome. Frameworks that already ship their own linting (NestJS always; `create-hono`'s packages without a config file) are detected and handled without creating a conflicting second setup.
4. **Docker** ‚Äî a multi-stage Node Dockerfile for servers, or a build-then-nginx Dockerfile for static frontends, plus a matching `docker-compose.yml`.
5. **Enterprise structure** ‚Äî the `.gitkeep`-tracked `src/` layout described above.

If a network step fails (e.g. installing Tailwind offline), the CLI records the dependency in `package.json`, finishes every filesystem step, and tells you exactly which command to run later ‚Äî it never leaves you with a half-broken directory and a stack trace. A few tools (`create-tauri-app`, Electron Forge, `create-hono`) always make some network install call of their own regardless of `--no-install`; the CLI warns about this rather than pretending it didn't happen.

### Known limitations

- **Tailwind/UnoCSS on Electron** isn't auto-wired ‚Äî Forge's template config layout varies too much to safely automate. A warning points you to the manual setup docs.
- **UnoCSS on Angular/Electron** and **Tailwind's `content` detection on Astro/Nuxt** follow the same "warn, don't guess" rule when a config shape can't be confidently matched.
- **`create-hono`'s `nodejs` template** currently pins a very new TypeScript version that `typescript-eslint` doesn't fully support yet ‚Äî this is an upstream compatibility issue in `create-hono` itself, not something this CLI causes, and it may intermittently fail `npm install` regardless of the tooling you pick here.
- **Desktop scaffolds (Tauri/Electron)** aren't build-tested in CI ‚Äî they need a Rust toolchain and Linux GUI libraries CI runners don't ship. CI verifies the scaffold and `package.json` land correctly; the native build itself is on you to verify locally.

## Releasing

Releases are fully automated with [semantic-release](https://semantic-release.gitbook.io/) via `.github/workflows/ci.yml`. On every push to `main`:

1. A representative matrix of stacks (across all five project types) is scaffolded, installed, linted, and built as a sanity check.
2. `semantic-release` analyzes commit messages since the last release using [Conventional Commits](https://www.conventionalcommits.org/):
   - `fix:` ‚Üí patch release
   - `feat:` ‚Üí minor release
   - `BREAKING CHANGE:` (in the commit body/footer) ‚Üí major release
   - `docs:`, `chore:`, `ci:`, etc. ‚Üí no release
3. If a release is warranted, it bumps `package.json`, updates `CHANGELOG.md`, publishes to **npm** and **GitHub Packages**, creates a Git tag + GitHub Release, and commits the version bump back to `main` (with `[skip ci]`).

This requires an `NPM_TOKEN` repository secret (Settings ‚Üí Secrets and variables ‚Üí Actions) ‚Äî a npm granular access token with **read/write** access to this package and **bypass 2FA** enabled.

### Manual publish (fallback)

```bash
npm login --registry=https://npm.pkg.github.com
npm publish --registry=https://npm.pkg.github.com   # GitHub Packages (publishConfig default)

npm login
npm publish --access public --registry=https://registry.npmjs.org   # public npm
```
